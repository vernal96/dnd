# Правила разработки для DnD Service

## Обязательные архитектурные принципы

- Весь новый код и все существенные переработки пишутся в парадигме DDD + Clean Architecture.
- Сервер является единственным источником истины.
- Клиент отправляет только intent-команды; валидация, применение правил и фиксация состояния происходят на сервере.
- Runtime, шаблоны сцен, persistent progression и encounter-слой должны оставаться разделенными.
- Критичный путь игрового действия должен быть синхронным и прозрачным: `command -> validate/apply -> persist -> publish event -> push to clients`.

## Требования к коду

- Писать максимально чистый и предсказуемый код с минимумом магии.
- Соблюдать строгую типизацию по умолчанию во всех новых и изменяемых PHP-файлах.
- Доменная модель, application-слой и инфраструктура должны быть разделены по ответственности и не смешиваться в одном месте без необходимости.
- Предпочитать явные имена, явные связи, явные таблицы и явные преобразования типов вместо неочевидных соглашений.
- Стараться избегать ассоциативных массивов в доменном и application-коде; вместо них использовать DTO.
- Исключение: задекларированные места Laravel, где массив является стандартным и ожидаемым контрактом фреймворка, например `config`, `casts`, `fillable`, фабрики, миграции, `response()->json()` и аналогичные точки интеграции.
- Стандартные файлы и шаблонные entrypoint-файлы Laravel без необходимости не изменять.
- Не прятать доменные правила в случайных сайд-эффектах ORM, клиента или инфраструктуры.
- Каждый метод обязан иметь PHPDoc с кратким описанием назначения.
- PHPDoc над методами писать на русском языке.
- Не использовать полные неймспейсы в коде и аннотациях, если тип можно импортировать через `use`.
- Не использовать `env()` в прикладном коде. В коде использовать `config()`, а в сидерах писать явные значения.
- Если сервисный метод может выбросить исключение, это нужно явно документировать через `@throws` на сервисном уровне.
- Исключения из сервисов нужно обрабатывать на более высоком уровне, например в контроллерах или application boundary, а не оставлять неявно всплывающими в HTTP-слой.
- Новый и изменяемый код должен быть совместим с проверками IDE без предупреждений и ошибок.
- Если соглашение фреймворка может быть понято двусмысленно или скрывает важную связь, поведение нужно задавать явно.

## Практика для Laravel-части

- Для моделей явно задавать нестандартные таблицы, ключи, casts и связи.
- Для моделей обязательно документировать поля через `@property` в PHPDoc класса, чтобы IDE корректно понимала атрибуты модели.
- Для контроллеров возвращать только подтвержденное сервером состояние.
- Для миграций описывать назначение `up()` и `down()`, а структуру таблиц держать близко к доменной модели из документов `docs/01_*`, `docs/02_*`, `docs/03_*`.
- Перед завершением работы прогонять как минимум синтаксическую проверку и тесты проекта, если среда это позволяет.
